  # Gorillas (Godot 4.5) Implementation Plan: Classic Core First, Procedural Skyline Later

  ## Summary

  Build a playable 2D local-hotseat artillery game inspired by QBASIC Gorillas with these locked decisions:

  1. Classic-first mechanics.
  2. Local hotseat only.
  3. Banana flight using RigidBody2D.
  4. Milestone 1 uses a fixed skyline.
  5. Explosions create simple crater damage in buildings.
  6. Input is numeric angle + velocity.
  7. Programmatic primitive visuals.
  8. Match winner is first to configurable hit count (default 3).
  9. Wind shown as arrow + number.

  Milestone 1 focuses on correct throw/explosion/hit loop and clean architecture so procedural skyline generation can be plugged in as Milestone 2.

  ## Architecture and Scene Composition

  1. Main.tscn (or Game.tscn) as root orchestrator.
  2. Child nodes:
  3. SkylineContainer (Node2D) to hold buildings.
  4. PlayerA and PlayerB gorilla nodes (Node2D or CharacterBody2D used as static actors).
  5. ProjectileLayer (Node2D) for banana spawn/lifecycle.
  6. EffectsLayer (Node2D) for explosion VFX.
  7. HUD (CanvasLayer) for angle/velocity input, turn state, score, wind indicator.
  8. Optional DebugOverlay for trajectory/physics diagnostics (toggleable).

  ## Core Systems

  1. GameController.gd
  2. Owns turn state machine, scoring, round resets, wind generation, win condition.
  3. Phases: ROUND_SETUP -> TURN_INPUT -> PROJECTILE_IN_FLIGHT -> RESOLUTION -> NEXT_TURN/ROUND_END -> MATCH_END.
  4. Prevents input while projectile is active.
  5. Emits state changes to HUD.
  6. PlayerController.gd
  7. Stores player id, score, spawn point, facing direction.
  8. Exposes request_throw(angle_deg, speed) called by GameController.
  9. BananaProjectile.gd (RigidBody2D)
  10. Spawned with initial linear velocity computed from angle/speed and player facing.
  11. Wind applied as continuous horizontal force (deterministic per frame from current turn wind value).
  12. Detects collisions with player/building/world bounds.
  13. On impact, emits projectile_impacted(position, normal, hit_target) then queues free.
  14. ExplosionSystem.gd
  15. Spawns visual explosion and applies crater damage at impact.
  16. Runs hit detection for gorillas within blast radius.
  17. Returns resolution result: no_hit, hit_player_a, hit_player_b, self_hit.
  18. Building.gd
  19. Represents single skyline block with destructible mask.
  20. Supports apply_crater(world_pos, radius):
  21. Updates collision shape approximation and visual mesh/sprite mask.
  22. Maintains simple damage model (circular subtraction).
  23. WindSystem.gd
  24. Generates per-turn wind scalar in configurable range (example -120..120 px/s^2 equivalent force mapping).
  25. Provides both numeric value and direction for HUD arrow.

  ## Public APIs / Interfaces / Types

  1. Enums/constants:
  2. TurnPhase enum in GameController.
  3. HitResult enum in ExplosionSystem.
  4. Data structures:
  5. ThrowCommand (angle_deg: float, speed: float).
  6. TurnContext (active_player_id: int, wind: float).
  7. MatchConfig resource:
  8. points_to_win: int = 3
  9. min_angle/max_angle, min_speed/max_speed
  10. wind_min/wind_max
  11. explosion_radius
  12. crater_radius
  13. Signals:
  14. GameController.turn_started(player_id, wind)
  15. GameController.turn_resolved(hit_result, scores)
  16. GameController.match_ended(winner_id)
  17. BananaProjectile.projectile_impacted(position, hit_target)
  18. Building.damaged() (for redraw/collider refresh)
  19. HUD.throw_submitted(angle, speed)

  ## Gameplay Rules (Milestone 1)

  1. Each turn: active player enters numeric angle + velocity, launches once.
  2. Wind is constant during that turn only.
  3. Banana follows RigidBody2D physics with added horizontal wind force.
  4. First collision triggers explosion and crater.
  5. If explosion radius overlaps opponent gorilla, shooter scores 1.
  6. If self-hit, opponent scores 1.
  7. Round resets actor/projectile positions and keeps cumulative score.
  8. Match ends when a player reaches points_to_win.

  ## Skyline Strategy

  1. Milestone 1: fixed skyline layout from static config array (building x/width/height values).
  2. Keep generator interface from day one:
  3. ISkylineProvider-style contract (implemented as script API):
  4. generate(seed) -> Array[BuildingSpec].
  5. Use FixedSkylineProvider now.
  6. Milestone 2 adds ProceduralSkylineProvider (random widths/heights/gaps, spawn-safe rooftop zones).

  ## UI/UX Plan

  1. HUD fields:
  2. Active player label.
  3. Numeric input for angle and velocity.
  4. Wind arrow + numeric strength.
  5. Scoreboard and target points.
  6. Launch button and input validation feedback.
  7. Disable controls during projectile flight and resolution.
  8. Keep restart/new match button on match end state.

  ## Testing and Validation

  1. Unit-style script tests (or lightweight runtime assertions):
  2. Angle/speed conversion yields expected initial velocity vectors.
  3. Wind mapping applies correct force direction and magnitude.
  4. Turn state machine forbids invalid transitions (no double launch).
  5. Score updates correctly on opponent hit and self-hit.
  6. Integration/manual scenarios:
  7. Flat no-wind throw hits expected parabola baseline.
  8. Positive wind and negative wind visibly alter horizontal drift.
  9. Projectile impacts building and crater appears at collision point.
  10. Consecutive hits reach win threshold and trigger match end UI.
  11. Invalid input (out of range/non-numeric) blocked with clear message.
  12. Turn alternation remains correct even after misses and self-hits.
  13. Acceptance criteria:
  14. Core loop is playable start-to-finish without editor intervention.
  15. At least one building can be cratered multiple times.
  16. Wind indicator matches observed drift direction.
  17. Match restart resets scores and scene state cleanly.

  ## Milestone Breakdown

  1. Milestone 1A: Scene scaffolding, turn state machine, HUD numeric input, player placement.
  2. Milestone 1B: RigidBody2D banana launch + wind + collision callbacks.
  3. Milestone 1C: Explosion VFX + crater damage + hit resolution/scoring.
  4. Milestone 1D: Match flow polish, restart flow, config resource tuning, test pass.
  5. Milestone 2: Swap fixed skyline provider with procedural generator while preserving interfaces.

  ## Assumptions and Defaults

  1. Engine: Godot 4.5 (from current project config).
  2. Desktop-first local play; no networking or AI in this plan.
  3. Primitive visuals are acceptable for v1.
  4. Crater damage is “simple” geometric subtraction, not structural physics.
  5. Determinism target is gameplay-consistent, not strict lockstep across platforms.
  6. Audio is optional for v1 and not required for acceptance.

• Buildings are a single bitmap that drives both visuals and collision.

  Flow in building.gd:

  1. Setup

  - size and building_color define dimensions/color (line 4:4, line 5:5).
  - On ready, _rebuild_bitmap() creates an Image filled with that color (line 15:15, line 47:47).

  2. Damage (crater)

  - apply_crater(world_pos, radius) converts explosion world position into building-local coordinates (line 24:24, line 25:25).
  - It loops pixels in a bounding box and clears pixels inside the circle (alpha = 0) (line 34:34 to line 41:41).
  - If anything changed, it refreshes render + collision (line 43:43 to line 45:45).

  3. Visual refresh

  - _refresh_visual_and_collision() uploads the modified Image into an ImageTexture, then assigns it to Sprite2D (line 52:52 to line 58:58).

  4. Collision refresh

  - It deletes old collision polygons (line 60:60 to line 61:61).
  - It converts opaque pixels in the image into polygons (BitMap.opaque_to_polygons) and creates new CollisionPolygon2D children (line 63:63 to line 71:71).

  Scene structure is minimal in Building.tscn: one Sprite2D for drawing and one StaticBody2D that gets generated polygon colliders.

  If helpful, I can also annotate this file with short comments directly in-code so it’s easier to maintain.
